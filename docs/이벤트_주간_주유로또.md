# 주간 주유로또 이벤트 기획안

> 상태: 초안(기획) · 작성 기준일 2026-06-24 · 구현 전 검토용
> 결정 완료 항목(사용자 확정): 당첨=**최다 일치+최초 우선**, 번호=**사용자 직접 선택**,
> 응모권=**1일 1장+주간 상한**, 산출물=본 문서.

---

## 1. 개요

순위(전국·지역·근처 TOP10)에 든 **싼 주유소**에서 주유 → **주유 기록 저장** → **리뷰 작성**까지
완료한 사용자에게 응모권을 주고, 매주 1명을 추첨해 **5만원권 주유 상품권**을 지급하는
참여형 이벤트.

- **목적/KPI**
  - 순위(저가) 주유소 실방문·실주유 유도 → 앱의 핵심 가치(최저가) 체감 강화
  - 주유 기록·리뷰 데이터 축적(리뷰 수, 기록 수 증가)
  - 주간 리텐션(매주 응모 동기) 향상
- **한 줄 정의**: "싼 주유소에서 넣고, 기록·리뷰 남기면 매주 1명 5만원 주유권."

---

## 2. 참여 조건 (자격)

한 장의 응모권은 아래 **3단계 자격 행동**을 모두 충족할 때 발급된다.

1. **순위 주유소에서 주유** — 지도에서 순위(전국/지역/근처 TOP10)에 표시된 주유소.
   회색 점("해골주유소" = 비하이라이트 일반 주유소)은 **제외**.
2. **주유 기록 저장** — 해당 주유소 상세에서 1클릭 주유 기록(`fuel_logs`) 저장.
3. **리뷰 작성** — 해당 주유소에 리뷰(`reviews`, 별점+내용) 작성/보유.

### 2.1 "순위 주유소"의 정의 (코드 기준)

| 구분 | 산정 방식 | 서버 검증 | 비고 |
|---|---|---|---|
| 전국 TOP10 | `queryNationalTop10(product)` — 전국 가격 오름차순 상위 10 | ✅ 결정적 | MVP 자격 |
| 지역 TOP10 | `queryRegionTop10(sido, product)` — 시·도별 상위 10 | ✅ 결정적 | v2 자격 |
| 근처 TOP10 | 사용자 GPS 반경(`queryStationsByRadius`) 상위 10 | ⚠️ GPS 의존 | v3(스푸핑 검토) |
| 회색 점(해골) | 위 어디에도 안 든 일반 주유소 | — | **자격 제외** |

> **권장**: MVP는 **전국 TOP10**만 자격으로 시작(서버에서 완전 재계산 가능 → 위변조 불가).
> 이후 **지역 TOP10**을 추가하고, **근처 TOP10**은 GPS 위변조 방지책 확정 후 도입.

### 2.2 리뷰 1회성과 반복 응모 (설계 포인트)

`reviews`는 **(사용자, 주유소)당 1건**(unique, 수정만)이라 같은 주유소로는 리뷰를 반복 작성할 수 없다.
따라서 응모권은 다음과 같이 정의한다.

- **응모권 = (회차 내) "순위 주유소에서의 주유 기록 1건" + "그 주유소 리뷰 보유"**.
- 같은 주유소를 재주유하면 기존 리뷰로 자격이 인정된다(매번 새 리뷰를 강요하지 않음).
- 결과적으로 **서로 다른 순위 주유소를 방문**할수록 응모 기회가 늘어 → 다양한 저가 주유소 방문 유도.

## 3. 응모 메커니즘

### 3.1 번호 선택 (사용자 직접)
- 응모 시 **1~25 중 서로 다른 5개**를 사용자가 직접 고른다(중복 불가).
- 응모권마다 번호 세트를 저장한다(한 사용자가 여러 응모권이면 각각 선택).

### 3.2 응모권 적립 단위·한도 (어뷰징 방지)
- **1일 1장**: 하루에 자격 행동을 여러 번 해도 응모권은 1장.
- **주간 상한 N장**(권장 **5장**): 한 회차(주)에 최대 N장까지.
- 한도 산정 기준은 **KST** 날짜/회차.

### 3.3 회차(주) 정의
- 1 회차 = **KST 월요일 00:00 ~ 금요일 24:00**.
- 응모 마감: 금요일 24:00. 추첨/발표: 토요일(21:00).

---

## 4. 추첨 & 당첨자 선정

- **추첨번호**: 매 회차 마감 후 **1~25 중 5개**를 발표.
  - MVP: 관리자가 추첨번호 입력(또는 시스템 RNG 후 관리자 확정).
  - v2 권장: **커밋-리빌(commit–reveal)** 방식 — 회차 시작 시 시드 해시를 미리 공개,
    마감 후 시드 공개로 번호를 검증 가능하게 생성(공정성 시비 차단).
- **당첨자 선정(매주 1명)**: 그 회차 응모권 전체에 대해
  1. **일치 개수**(응모 번호 5개 ∩ 추첨 번호 5개) **내림차순**,
  2. 동점이면 **응모 시각(entered_at) 빠른 순**.
  - → **1위 응모권 1장의 소유자**가 당첨. 매주 반드시 1명 나온다(응모가 1건 이상이면).
- 동일인 다중 응모권이 1위여도 **사람 기준 1명/회차**(중복 수상 없음, 차순위 승계 정책 명시).

## 5. 보상

- **5만원권 주유 상품권**(주유 전용 모바일/지류 상품권). 조달처는 운영 결정(§13).
- **지급 흐름**: 당첨 통보(푸시+이메일) → 본인 확인(연락처) → 상품권 코드 전달.
- **수령 기한**: 통보 후 7일. 미수령/연락 두절 시 **차순위 승계** 또는 소멸(약관 명시).
- 부정 참여 적발 시 **당첨 무효 + 향후 참여 제한**.

## 6. 어뷰징 방지 (핵심 리스크)

주유 기록은 **자기신고(1클릭, GPS 무관)** 라 허위 생성이 가능하다. 다층 방어를 둔다.

1. **자격 주유소를 서버 검증 가능한 순위로 한정**(전국/지역 TOP10) → 임의 주유소로는 응모 불가.
2. **1일 1장 + 주간 상한** → 대량 생성 차단.
3. **리뷰 품질 게이트**: 최소 글자수, (선택) 사진 1장 이상, 도배/복붙 필터, `is_hidden` 모더레이션.
4. **계정·디바이스 중복 방지**: 가입 이력 + **전화번호 인증**(0017/0018 인프라 활용) 권장.
5. **기록 신뢰도 보강(선택)**: 주유 시 GPS 동의 좌표를 받아 주유소 1~2km 이내 스냅샷 저장(근처 자격/검증용).
6. **당첨 후 본인 확인** 후에만 지급. 이상 패턴(단시간 대량·동일 IP/디바이스)은 수기 검수.

## 7. 자격 검증 설계 (기술)

응모권 발급 시 **서버가 순위를 재계산**해 자격을 판정한다(클라이언트 신뢰 금지).

```
POST /api/event/entry  (자격 행동 완료 후 호출)
 ├─ 1) 인증 사용자 확인 + 회차(open) 확인
 ├─ 2) 한도 검사: 오늘 발급 0장 && 이번 회차 < 주간상한
 ├─ 3) 주유 기록 존재: fuel_logs (user, station, logged_at ∈ 회차)
 ├─ 4) 리뷰 보유: reviews (user, station)
 ├─ 5) 순위 검증(재계산):
 │      station ∈ queryNationalTop10(product)        ... 전국
 │      또는 station ∈ queryRegionTop10(sido, product) ... 지역(v2)
 ├─ 6) 번호 검증: 1~25 distinct 5개
 └─ 7) event_entries insert (numbers, qualified_rank_type, fuel_log_id, review_id)
```

- **가격 스냅샷 주의**: `prices_latest`는 동기화로 갱신되므로 "주유 시점 순위"와 "발급 시점 순위"가
  다를 수 있다. 단순화: **발급 시점 당일 순위**로 판정하고, 회차 집계 시 1회 재검증(이상치 제거).
- **근처(v3)**: 주유 시 저장한 GPS 스냅샷으로 `queryStationsByRadius` 재계산 → 포함 검사.
  GPS 스푸핑 한계가 있어 한도·본인확인과 병행.

## 8. 데이터 모델 (신규 마이그레이션 `0031_event_weekly_lotto.sql` 초안)

```sql
-- 회차(주) 정의 + 추첨 결과
create table if not exists event_draws (
  id              uuid primary key default gen_random_uuid(),
  week_start      date not null,                 -- KST 월요일
  week_end        date not null,                 -- KST 일요일
  status          text not null default 'open',  -- open | drawn | closed
  winning_numbers smallint[],                    -- 추첨 5개(1~25). 추첨 전 null
  drawn_at        timestamptz,
  winner_entry_id uuid,                           -- 당첨 응모권(집계 후)
  created_at      timestamptz not null default now()
);
create unique index if not exists event_draws_week_uidx on event_draws (week_start);

-- 응모권
create table if not exists event_entries (
  id             uuid primary key default gen_random_uuid(),
  draw_id        uuid not null references event_draws(id) on delete cascade,
  user_id        uuid not null references users(id) on delete cascade,
  station_id     text not null,
  station_name   text not null,
  numbers        smallint[] not null,            -- 사용자가 고른 5개(1~25 distinct)
  rank_type      text not null,                  -- national | region | nearby
  fuel_log_id    uuid,                           -- 근거 주유 기록
  review_id      uuid,                           -- 근거 리뷰
  entry_date     date not null,                  -- KST(1일 1장 키)
  match_count    smallint,                        -- 집계 시 채움
  entered_at     timestamptz not null default now()
);
-- 1일 1장
create unique index if not exists event_entries_user_day_uidx
  on event_entries (user_id, draw_id, entry_date);
-- 당첨자 산출(일치수 desc, 응모시각 asc)
create index if not exists event_entries_draw_rank_idx
  on event_entries (draw_id, match_count desc, entered_at asc);

alter table event_draws  disable row level security;
alter table event_entries disable row level security;
```

> 주간 상한 N은 앱 로직(발급 전 count)으로 강제(유니크 인덱스는 1일 1장만 보장).

## 9. API / 화면

### 9.1 사용자
- **응모 동선**: 순위 주유소 상세(순위 배지 노출) → 주유 기록 + 리뷰 작성 후 **"이벤트 응모"** CTA
  → 1~25 중 5개 선택 → 응모 완료.
- **내 응모 현황**: 이번 회차 응모권/번호, 잔여 응모 가능 수, 지난 회차 결과·당첨 여부.
- **결과 화면**: 회차 추첨번호 + 내 일치 개수, 당첨 안내.

### 9.2 서버 API
- `POST /api/event/entry` — 자격 검증 + 응모권 발급(§7).
- `GET  /api/event/current` — 현재 회차/마감/지난 추첨번호.
- `GET  /api/event/me` — 내 응모/결과.

### 9.3 관리자 (`/admin/event`, ADMIN_EMAILS 게이트 — 공지 관리와 동일 패턴)
- 회차 생성/마감, 추첨번호 입력 또는 RNG 추첨.
- **당첨자 산출**(일치수 desc, 응모시각 asc) + 지급 처리(상품권 코드/상태).
- 응모 통계·어뷰징 모니터링(사용자별/디바이스별 집계).

## 10. 운영 플로우 (주간 사이클)

```
월 11:00  지난 회차 추첨번호 발표 → 당첨자 산출 → 통보(푸시+이메일) → 새 회차 open
화~토     응모 누적(자격 행동 → 응모권 발급)
일 24:00  응모 마감(회차 status=drawn 대기)
다음 월   상품권 지급 + 본인확인 마감(7일 내)
```

알림은 기존 **web-push(VAPID, 0027)** + 이메일을 활용.

## 11. 법적 / 정책 검토 (⚠️ 전문가 확인 필요)

- **경품 규제**: 「경품류 제공에 관한 불공정거래행위의 유형 및 기준」상 **소비자현상경품(추첨)** 한도 준수.
  5만원권은 통상 문제 범위는 아니나, **이벤트 약관·유의사항 고지** 필수.
- **세금(기타소득)**: 경품은 기타소득. 소득세법상 **건별 5만원 이하**는 과세최저한(원천징수 면제) 경계.
  → 제세공과금/신고 부담을 줄이려면 상품권 가액을 **5만원 이하**로 설계하거나 제세공과금 처리 정책 명시.
  (정확한 과세 판단은 세무 검토 필요 — 본 문서는 면책.)
- **개인정보**: 당첨자 연락처·지급정보 **최소 수집·동의**, 보관기간 한정.
- **약관 페이지**: 부정참여 무효, 지급조건, 차순위 승계, 변경/중단권, 미성년자 정책 명시.

## 12. 단계별 구현 범위

| 단계 | 범위 |
|---|---|
| **MVP (1~2주)** | 전국 TOP10 자격 · 사용자 번호선택 · 1일1장/주간상한 · 응모/현황 화면 · 관리자 수동 추첨·당첨산출·수기 지급 |
| **v2** | 지역 TOP10 자격 추가 · 커밋-리빌 RNG 추첨 · 자동 당첨 산출 · 푸시 통보 · 상품권 지급 흐름 정형화 |
| **v3** | 근처 TOP10(GPS 검증) · 리더보드/연속 응모 보상 등 재미 요소 |

## 13. 미해결 / 결정 필요

1. **상품권 조달처**: 주유 상품권 vendor(GS/SK/모바일권)와 단가·정산·코드 발급 방식.
2. **추첨 공정성**: MVP 관리자 수동 vs 처음부터 커밋-리빌 RNG.
3. **번호 선택 단위**: 응모권마다 선택(현 안) vs 사용자 고정 "행운번호" 1세트 적용.
4. **근처 TOP10 포함 여부**: GPS 스푸핑 허용 위험 vs 참여 폭.
5. **주간 상한 N 값**: 권장 5(조정 가능).
6. **자격 가격 스냅샷 정책**: 주유 시점 vs 발급 시점 순위(현 안: 발급 시점 당일).
7. **본인확인 강도**: 전화 인증 의무화 여부(어뷰징 vs 진입장벽).
